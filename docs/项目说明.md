# Blast-Wave 项目说明

## 这是什么

这是一个用 C++17 和 ROOT 实现的小型 Blast-Wave 事例生成器。它的目标不是做完整重离子末态模拟，而是在一个足够小、足够清晰、可继续扩展的代码骨架里，完成以下事情：

- 用 Monte Carlo Glauber 产生逐事例的 participant 几何涨落
- 在 participant hotspot 上做简化的 blast-wave 发射
- 通过 ROOT 输出事件树、粒子树和基础 QA 直方图
- 提供一个独立 QA 程序检查输出是否合法、接口是否稳定

当前第一版只生成单一直接 `pi+`，适合做几何和流场方向的快速验证，不适合直接拿去和实验末态全带电粒子分布逐项对比。

## 当前能力边界

已经支持：

- 固定冲量参数 `b` 的事件生成
- 由 `b` 线性近似得到的事件级 `centrality`
- Pb-Pb 风格 Woods-Saxon 核几何
- participant eccentricity `eps2` 和 participant plane `psi2`
- 局域热动量抽样
- 局域流场 + 正确 Lorentz boost
- ROOT 输出与独立 QA
- CMake 构建

还没有做：

- 最小偏置 `b` 抽样
- 共振衰变
- 多粒子种类
- hadronic rescattering
- 完整 Cooper-Frye freeze-out 超曲面采样

## 目录结构

- `include/blastwave/BlastWaveGenerator.h`
  对外数据结构和生成器类声明。
- `src/BlastWaveGenerator.cpp`
  核几何、participant 形状、发射源、流场、boost 和内部校验。
- `apps/generate_blastwave_events.cpp`
  命令行生成程序，负责写 ROOT 文件。
- `apps/qa_blastwave_output.cpp`
  独立 QA 程序，负责读取 ROOT 文件并做一致性检查。
- `CMakeLists.txt`
  主构建文件。
- `CMakePresets.json`
  默认 CMake preset，同时开启 `compile_commands.json` 输出。
- `qa/`
  本地测试输出样本。
- `docs/`
  设计、handoff 和说明文档。
- `qiye/`
  历史参考宏，只用于理解物理意图，不应直接复制代码。

## 如何编译

本项目默认使用当前机器上的 O2Physics/ROOT 环境来编译。

在项目根目录执行：

```bash
/bin/zsh -lc "alienv setenv O2Physics/latest-master-o2 -c sh -lc 'cmake --preset default -S /Users/allenzhou/Research_software/Blast_wave'"
/bin/zsh -lc "alienv setenv O2Physics/latest-master-o2 -c sh -lc 'cmake --build /Users/allenzhou/Research_software/Blast_wave/build'"
```

构建完成后会得到：

- `bin/generate_blastwave_events`
- `bin/qa_blastwave_output`
- `build/compile_commands.json`

其中 `compile_commands.json` 已经开启，便于编辑器、clangd 和静态分析工具使用。

## 运行参数

生成器程序 `generate_blastwave_events` 现在支持三种等价入口：

- `generate_blastwave_events [options]`
- `generate_blastwave_events --config <path> [options]`
- `generate_blastwave_events <config-path> [options]`

优先级固定为：

- 显式 CLI 参数
- 配置文件中的参数
- `BlastWaveConfig` 内建默认值

配置文件使用轻量 `key = value` 文本格式：

- 允许空行
- 允许整行 `#` 注释
- 支持的键名与当前公开 CLI 参数一致：
  - `nevents`
  - `b`
  - `temperature`
  - `tau0`
  - `smear`
  - `sigma-nn`
  - `seed`
  - `output`
  - `vmax`
  - `kappa2`
  - `sigma-eta`
  - `eta-plateau`
  - `nbd-mu`
  - `nbd-k`
  - `r-ref`
- 配置文件里的相对 `output` 路径会相对配置文件所在目录解析

最小示例：

```text
# qa/test_b8.cfg
nevents = 5000
b = 8
temperature = 0.2
tau0 = 10.0
smear = 0.5
output = test_b8_from_config.root
```

以下参数仍然可以继续直接通过 CLI 传入。

基础参数：

- `--nevents <int>`
  生成多少个事件。
- `--b <fm>`
  冲量参数，单位 `fm`。
- `--temperature <GeV>`
  局域静止系热动量抽样的温度尺度。
- `--tau0 <fm/c>`
  固定冻结 proper time。
- `--smear <fm>`
  participant hotspot 的横向高斯展宽。
- `--sigma-nn <fm^2>`
  核子-核子非弹截面参数，默认 `7.0 fm^2`。
- `--seed <uint64>`
  随机数种子。
- `--output <path>`
  输出 ROOT 文件路径。

调参与 QA 相关参数：

- `--vmax <value>`
  最大横向流强度上限。
- `--kappa2 <value>`
  二阶各向异性响应强度。
- `--sigma-eta <value>`
  纵向 `eta_s` 高斯尾宽度。
- `--eta-plateau <value>`
  纵向 `eta_s` 平顶核心半宽。
- `--nbd-mu <value>`
  每个 hotspot 的平均发射强度。
- `--nbd-k <value>`
  NBD 形状参数，控制涨落强弱。
- `--r-ref <fm>`
  流场径向 profile 的参考半径。

QA 程序 `qa_blastwave_output` 的参数较少：

- `--input <file.root>`
- `--output <qa.root>`
- `--expect-nevents <int>`

核心命名空间是 `blastwave`。

保存生成器参数。当前默认值主要包括：

- `nEvents = 100`

- `charge = 1`

- `tau0 = 10.0 fm/c`
- `smearSigma = 0.5 fm`
- `sigmaNN = 7.0 fm^2`

- `nbdMu = 2.0`
- `nbdK = 1.5`

- `kappa2 = 0.5`
- `woodsSaxonRadius = 6.62 fm`
- `woodsSaxonDiffuseness = 0.546 fm`
- `mass = 0.13957 GeV`

### `EventInfo`

每个事例的摘要信息：

- `eventId`
- `impactParameter`
- `nParticipants`
- `eps2`
- `psi2`
- `nCharged`

### `ParticleRecord`

每个粒子的记录：

- 事件映射信息
- 粒子身份信息
- 发射坐标 `x, y, z, t`
- 四动量 `px, py, pz, energy`
- 源位置信息 `etaS, sourceX, sourceY`

### `GeneratedEvent`

由一个 `EventInfo` 和一个粒子数组组成。

### `BlastWaveGenerator`

核心入口：

- 构造：`BlastWaveGenerator(BlastWaveConfig config)`
- 生成单事例：`generateEvent(int eventId)`

这部分核心代码不依赖 ROOT，ROOT 只在应用层负责文件写出和 QA。

## ROOT 输出文件里有什么

一个生成结果 ROOT 文件包含三棵树和若干 QA 直方图。

### `events` 树

分支：

- `event_id`
- `b`
- `Npart`
- `eps2`
- `psi2`
- `centrality`
- `Nch`

### `participants` 树

分支：

- `event_id`
- `nucleus_id`
- `x`
- `y`
- `z`

### `particles` 树

分支：

- `event_id`
- `pid`
- `charge`
- `mass`
- `x`
- `y`
- `z`
- `t`
- `px`
- `py`
- `pz`
- `E`
- `eta_s`
- `source_x`
- `source_y`

### 同文件写出的直方图

- `Npart`
- `eps2`
- `psi2`
- `cent`
- `participant_x-y`
- `x-y`
- `px-py`
- `pT`
- `eta`
- `phi`

其中：

- `participant_x-y` 用来看 participant 核子在横向平面的分布
- `x-y` 用来看空间发射分布
- `px-py` 用来看横向动量分布
- `psi2` 是 participant plane 的角分布
- `cent` 是按 10% 分档、范围到 `110%` 的中心度直方图；当前实现只填 `0-100`

此外，输出文件里还会写一个 `participant_x-y_canvas`。这是一张已经画好的 ROOT 画布，会把 `participant_x-y` 直方图和核 A、核 B 的圆形轮廓一起存进去，便于直接查看 participant 和几何 overlap 的关系。

QA 程序会要求这些对象全部存在，否则直接报错。

## 算法是怎么做的

### 1. 几何阶段

先对两个核分别做 Woods-Saxon 抽样，得到 3D 核子位置。然后把两个核沿 `x` 方向平移 `±b/2`。接着只在横向平面上判断核子是否发生碰撞：如果两核子的横向距离满足

`dx^2 + dy^2 <= sigmaNN / pi`

就把它们标记为 participant。最后把两边所有 participant 合并成该事例的参与核子集合。

### 2. `eps2` 和 `psi2`

先计算 participant 在横向平面的质心，再计算中心化后的二阶矩：

- `sigmaX2`
- `sigmaY2`
- `sigmaXY`

然后构造二阶偏心率向量：

- `eccentricityX = sigmaY2 - sigmaX2`
- `eccentricityY = 2 * sigmaXY`

最终得到：

- `eps2 = sqrt(ex^2 + ey^2) / (sigmaX2 + sigmaY2)`
- `psi2 = 0.5 * atan2(ey, ex)`

物理含义是：

- `eps2` 表示横向几何有多“扁”
- `psi2` 表示这个椭圆主轴朝哪个方向

### 3. 每个 participant 发多少粒子

每个 participant 被视为一个发射 hotspot。其发射粒子数不是固定值，而是用一个 Gamma-Poisson 混合模型来抽样，也就是常说的 NBD-like 涨落模型。这样可以保留逐热点、逐事例的粒子数涨落。

### 4. 发射点坐标

每个粒子并不从 participant 的精确位置发出，而是先对横向位置做一个宽度为 `smearSigma` 的高斯展宽。纵向部分采用固定 `tau0`，再结合 `eta_s` 得到：

- `t = tau0 * cosh(eta_s)`
- `z = tau0 * sinh(eta_s)`

### 5. 纵向分布

当前实现已经不是“纯高斯 `eta_s`”，而是：

- 中心区是平顶核心
- 两侧是高斯尾部

对应参数是：

- `etaPlateauHalfWidth`
- `sigmaEta`

这样做的原因是，纯高斯源在中快度会产生过尖的 `eta` 分布；平顶核心更接近当前项目想要的近中快度平台行为。

需要注意，这仍然只是一个现象学近似，不是完整的实验 `dN/deta` 描述。

### 6. 局域热动量抽样

代码先在流体局域静止系里抽样粒子的热动量。当前用的是一个轻量近似：

- 动量大小从 Gamma 分布抽样
- 方向各向同性
- 能量由壳条件 `E^2 = p^2 + m^2` 给出

它的优点是实现简单、稳定，适合当前 v1。缺点是它还不是严格的 Cooper-Frye 采样。

### 7. 流场

横向流场依赖粒子发射点的半径和方位角，同时引入 `eps2`、`psi2` 控制的二阶调制：

- `modulation = 1 + 2 * kappa2 * eps2 * cos(2 * (phi - psi2))`
- `profile = clamp((r / referenceRadius) * modulation, 0, 1)`
- `vT = clamp(vMax * profile, 0, 0.95)`

再把这个横向流与 `eta_s` 方向上的 Bjorken-like 纵向流组合，构造出局域速度场。

### 8. 正确 boost 到实验室系

这是该项目和旧参考思路最重要的区别之一。当前实现不是简单地给 `px`、`py` 人工加一个偏移量，而是：

1. 先在局域静止系抽样热动量
2. 再按局域流速做完整 Lorentz boost

这比直接“加速度偏移”更符合 blast-wave 的物理过程。

## QA 程序会检查什么

`qa_blastwave_output` 会独立打开 ROOT 文件并检查：

- `events`、`participants` 和 `particles` 三棵树是否存在
- 必需直方图是否存在
- 事件号是否连续
- `Npart` 是否等于该事件在 `participants` 树中的实际条目数
- `Nch` 是否等于该事件在粒子树中的实际条目数
- `centrality` 是否保持在 `0-100` 且与当前 `b` 的线性映射一致
- 是否出现 `NaN` 或 `Inf`
- 所有粒子是否满足壳条件，误差不超过 `1e-4 GeV^2`

同时它会把重新读出的 QA 直方图写到新文件中，例如：

- `qa_x-y`
- `qa_px-py`
- `qa_psi2`
- `qa_pT`
- `qa_eta`
- `qa_phi`

## 几个容易误解的点

### `eps2` 到底是什么

它是二阶 participant eccentricity，也就是初态参与核子横向分布的椭圆偏心率。它不是末态流系数 `v2`，而是一个“初态几何各向异性”的量。

### 为什么 `px-py` 不一定肉眼上就像横椭圆

因为原始 inclusive `px-py` 图会被大量低 `pT` 的近各向同性热粒子主导，所以即便存在正确方向的二阶各向异性，2D 密度图看起来也可能不明显。更敏感的量通常是：

- `phi` 分布
- `v2`
- 对较高 `pT` 做切后再看动量各向异性

### 为什么 `eta` 不能直接和实验 `dN_ch/deta` 一一对应

因为当前模型只生成单一直接 `pi+`，没有：

- 其他粒子种类
- 共振衰变 feed-down
- 探测器接受度和效率

所以这里的 `eta` 更适合做模型内部趋势检查，而不是直接拿来当实验全带电分布。

## 当前实现上的注意事项

- ROOT 树里的浮点分支现在使用 `Double_t`，不是 `Float_t`
  原因是之前 `Float_t` 在大 `|eta_s|` 尾部会让壳条件偏差变大，不利于 QA。
- 生成器核心尽量保持 ROOT-free
  如果后续要扩展算法，优先改 `include/` 和 `src/`，不要把 ROOT 类型带进核心物理层。
- 输出文件接口现在已经比较稳定
  如果修改树名、分支名或直方图名，需要同步更新 QA 程序和所有下游分析。

## 如果后续继续扩展，优先建议做什么

按性价比排序，比较值得优先做的是：

1. 增加更直接的各向异性观测量，例如 `phi - psi2` 和 `v2(pT)`
2. 增加多粒子种类与共振衰变
3. 把当前简化热抽样升级为更标准的源函数或 Cooper-Frye 采样
4. 支持最小偏置 `b` 抽样和中心度切分

## 一句话总结

这个项目当前最适合被理解为：一个结构清晰、接口稳定、便于继续演化的 v1 blast-wave 生成器骨架，用来研究 Glauber 几何如何通过局域流场映射到末态单粒子相空间分布。
